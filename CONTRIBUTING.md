## Contributing

As things are changing quickly, and this is a small project maintained by one person with a full-time job, these guidelines have been added to establish some ground rules for contributions. 

The maintainers reserve the right to make calls that are intended to honor the spirit of these guidelines as the situation continues to evolve, even if not directly spelled out in these rules.

### AI Policy

This project doesn't outright ban contributions that are AI-assisted. However, software engineering principles still apply. As computers cannot be held accountable or liable, all changes and issues must be coming from a person who can be responsible for their contribution.

Any contribution that is fully authored by an AI agent will not be accepted. Delegating interactions with the maintainers to an AI agent will cause your contribution to not be accepted. There _MUST_ be a person behind all contributions for the purposes of engineering discussion, and code ownership. 

Any issues or pull requests submitted in violation of the above rules will be closed. Suspected infractions of the above rules will be handled at the sole discretion of the maintainers. 

### Submitting Issues

Please open an issue using one of the provided templates. If your issue doesn't match one of the existing templates, use the `Feature request` template.

After some initial discussion and investigation, issues will be tagged as `accepted`. This means they are now open for the maintainers or contributors to provide pull requests against that issue. This doesn't guarantee a timeframe for when the issue will be closed out with a change. 

### Submitting Pull Requests

Before submitting a pull request, ensure there's an existing issue, and that it has been marked `accepted` by a maintainer. If there isn't one, open one explaining the change you want to make and why. It doesn't need to be complex, it just needs to properly describe the change. This issue will serve as the record of design discussion around the change, and help catch design concerns early that might block acceptance of your pull request.

Pull requests shall be single-issue, and reference the issue that they are resolving in the summary. If there isn't an existing issue, or the issue hasn't been tagged `accepted`, then your pull request will be rejected. 

Any discussion of a rejected implementation will continue in the issue raised for it.
